# PRD 反审查报告（Karpathy 视角）

> 审查方式：用 Karpathy 守则（少假设、求简洁、精准改动、目标驱动）反向审视
> 关联文档：`PRD-登录与CDK.md`、`PRD-登录与CDK-审查报告.md`
> 输出由子 agent 生成，未做改动
> 核心立场：第一轮审查是「找漏洞、加防御」视角；本轮是「反过度设计、砍臆测」视角

---

## TL;DR

PRD 在「臆测性功能」(多设备控制、密码找回独立流程、session 列表 / 单删 API、续费上限) 上明显过度；审查报告又叠加了一层「为不可能场景做防御」(账号枚举、限频字典 GC、登录锁定)。砍到极简后，数据库可从 4 张表 → 2 张表，API 从 11 个 → 5 个，实现工时可从 9-10 天 → 4-5 天。**核心病灶：把咸鱼月销几十单的小生意当 SaaS 在做。**

---

## A. PRD 中的过度设计

### 问题 A1：`password_resets` 表 + 独立找回密码流程
- **判断**：过度
- **守则**：第 2 条「不写用户没要求的功能」「单次使用的代码不抽象」
- **理由**：用户全在咸鱼一对一沟通，密码丢了直接私聊「我帮你重置」更直接。月销几十单意味着一年触发找回密码可能 < 10 次，为此搭 SMTP + 邮件模板 + token 表 + 两个接口 + 一个 HTML 页，典型「200 行代码做 50 行的事」。
- **建议**：**一期砍掉**。卖家手工在 SQLite 里 `UPDATE users SET password_hash=? WHERE email=?` 即可。整张表、两个接口、`reset.html`、QQ SMTP 配置全部移除。

### 问题 A2：多设备控制（3 session 上限 + 列表 + 踢下线）
- **判断**：过度
- **守则**：第 1 条「必要时反推用户的需求」+ 第 2 条「不擅自加灵活性」
- **理由**：用户原话只是「PC 看完手机也能看」=「同账号能在多端登录」，而不是「限制到 3 台」。「3」是 PRD 自己拍的猜测。`/me/sessions` GET/DELETE 是为这个臆想需求陪葬的。
- **建议**：**一期砍掉 session 上限**。改用无状态 JWT（15 天有效期），sessions 表整张删除，登出在前端清 localStorage 即可。`/me/sessions` 两个接口删除。多账号共用真出问题再做。

### 问题 A3：bcrypt cost=12
- **判断**：过度
- **守则**：第 2 条「资深工程师会觉得这段过度复杂吗」
- **理由**：cost=12 比 cost=10 慢 4 倍（~250ms vs ~60ms），对一个 2 核 2G 的小机器、月销几十单的场景毫无意义，只会让登录接口在 CPU 紧张时排队。
- **建议**：cost=10。或直接用 `passlib` 默认值不指定。

### 问题 A4：IP 限频 + 账号限频双层
- **判断**：过度
- **守则**：第 2 条「不为不可能发生的错误场景写处理」
- **理由**：咸鱼小站不是攻击目标。账号级 60/分已足够防滥用爬；IP 级再叠一层只是增加进程内字典维护负担。
- **建议**：**只保留账号级限频**。`/login` `/register` 不限频（反正攻击者绕 IP 很容易，真要防爆破需要登录失败计数，见 B 区，也建议不做）。

### 问题 A5：CDK 命令行工具用 argparse + `--count --days`
- **判断**：过度
- **守则**：第 2 条「单次使用的代码不抽象」「不擅自加可配置性」
- **理由**：duration 写死 30 天，count 一次生成 50，**写死参数 5 行代码搞定**。argparse + 帮助文档是给「会被多人长期使用的工具」准备的。
- **建议**：`tools/gen_cdk.py` 直接 `for _ in range(50): print(generate())`，要改数量改源码。

### 问题 A6：JWT vs sessions 表的默认选择
- **判断**：过度（决策本身基于一个不必要的需求）
- **守则**：第 1 条「不要默默选一种解释」「列出 alternatives」
- **理由**：PRD 第 110 行「JWT 无状态，做不了踢下线」直接否决了 JWT——但「踢下线」本身就是 A2 里被砍掉的功能。前提消失，结论应反转。
- **建议**：**改用 JWT**（`PyJWT`，HS256，15 天）。sessions 表整张删，无并发竞态、无 GC、无内存增长。

### 问题 A7：11 个 API 接口
- **判断**：过度
- **守则**：第 2 条「最少代码」
- **理由**：砍掉 A1+A2 后，`/password-reset/*`(2)、`/me/sessions`(2)、`/logout`(1，JWT 客户端清掉就行) 共 5 个直接消失。
- **建议**：留 5 个：`/register` `/login` `/redeem` `/me` `/daily/*`。

## B. 审查报告中的过度防御

### 问题 B1：续费上限 90 天「防黑产囤 100 个 CDK」
- **判断**：过度
- **守则**：第 2 条「不为不可能发生的错误场景写处理」
- **理由**：卖家自己手工发 CDK，谁会一次给一个用户发 100 个？这是把 SaaS 风控搬到咸鱼。真实场景：用户买完续费 1-2 个 CDK 就回来，8 年后到期对你毫无威胁。
- **建议**：**忽略此建议**，续费公式不加上限。

### 问题 B2：账号枚举漏洞「统一返回模糊错误」
- **判断**：过度
- **守则**：第 2 条「不为不可能发生的错误场景」
- **理由**：这是高价值目标（银行 / SaaS）才需要的。咸鱼新闻站知道某邮箱注册了又如何？用户体验比模糊文案重要——「该邮箱已被使用」可以让用户立刻明白要去登录。
- **建议**：**忽略**，正常返回明确错误。

### 问题 B3：session 被踢的并发竞态
- **判断**：过度（已被 A2 消除）
- **守则**：第 2 条「不可能发生的错误场景」
- **理由**：砍掉多设备上限 + 改 JWT 后，这个问题根本不存在。
- **建议**：无需处理。

### 问题 B4：限频字典内存泄漏 + 定时 GC
- **判断**：过度
- **守则**：第 2 条「不为不可能发生的错误场景写处理」
- **理由**：数量级估算：月销几十单 → 几十个活跃用户 → 几十个 IP/天 → 字典每条 deque 几十字节 → 一年累计 ~365×100×100B ≈ 3.6MB。**单 worker 跑 5 年都撑得住**。GC 逻辑是给百万 QPS 服务的。
- **建议**：**忽略**，字典自然增长，服务每次重启自然清零。

### 问题 B5：数据采集失败时旧数据如何对外
- **判断**：过度
- **守则**：第 1 条「反推用户需求」
- **理由**：前端默认显示 `index.json` 里最新一天即可。crontab 没跑成功，昨天数据还在，用户多看一天旧新闻不会投诉。「区分新旧」是 PRD 自己加的需求。
- **建议**：**忽略**，前端不区分。

### 问题 B6：首次部署「鸡生蛋」
- **判断**：合理但越界
- **守则**：第 3 条「精准改动」——这是运维 runbook，不是 PRD 范围
- **理由**：确实需要，但不应塞进 PRD。
- **建议**：开工时单独写一份 `DEPLOY.md`，PRD 不收录。

### 问题 B7：审查报告自加的「登录失败锁定 + systemd OnFailure」
- **判断**：过度（报告本身越权增加 PRD 没要求的功能）
- **守则**：第 2 条「不写用户没要求的功能」
- **理由**：PRD 没要求登录锁定，审查报告以「建议」名义把它塞进来。咸鱼场景没有暴力破解威胁。systemd OnFailure 同理——服务挂了卖家自己访问会发现，不需要钩子。
- **建议**：**全部忽略**。

### 问题 B8：WAL + `sqlite3 .backup` 替代 `cp`
- **判断**：合理（保留）
- **理由**：这条是真问题，代价小、收益明确，符合「精准改动」。

### 问题 B9：外键 `PRAGMA foreign_keys=ON`
- **判断**：合理（保留）
- **理由**：一行代码避免静默 bug，符合「最小修复」。

## C. 未摊开的假设

### 问题 C1：邮箱+用户名+密码三件套
- **判断**：过度
- **守则**：第 1 条「列出 alternatives」+ 第 2 条「最少」
- **理由**：邮箱已经唯一，用户名是冗余字段。砍掉用户名：登录只用邮箱、`users` 表少一列、注册表单少一字段、唯一性校验少一处。
- **建议**：**只用邮箱**。

### 问题 C2：「FastAPI + SQLite + Nginx + 腾讯云 2H2G + 备案」是默认选择
- **判断**：不确定（取决于卖家是否在意备案 7-20 天等待）
- **守则**：第 1 条「把权衡摊到台面上」
- **理由**：PRD 没讨论 Cloudflare Workers + D1（零备案、零运维、月销几十单完全在免费额度）、Vercel + Turso、香港轻量（免备案）等替代方案。「腾讯云 2H2G + 备案」是默认选的，没说为什么。
- **建议**：PRD 应补「方案对比表」列 2-3 个替代方案，显式选择并写理由。**最简方案：香港轻量 + SQLite，跳过备案，9-10 天 → 4-5 天可上线**。

### 问题 C3：备份策略「每天 cp」vs「不备份」
- **判断**：过度（对新闻站）
- **守则**：第 2 条「最少」
- **理由**：用户表丢了顶多让用户重新拿 CDK 兑换；CDK 表丢了卖家重新生成 50 个发给老客户；新闻数据丢了第二天采集自动重生。**这个站没什么数据需要保护**。
- **建议**：**一期不备份**。SQLite 真挂了，一份「重新初始化 + 通知用户重新注册」的恢复脚本就够。

## D. 验收标准缺失

### 问题 D1：M1-M6 每个里程碑「完成」标志未定义
- **判断**：过度（指 PRD 漏了关键内容）
- **守则**：第 4 条「先定义成功标准」「强成功标准让你独立闭环」
- **理由**：「2 天完成 FastAPI 骨架」——什么算完成？跑起来了？有 API doc？有测试？没标准就会无限拖延或敷衍交差。
- **建议**：每个里程碑写 1 条 e2e 验证命令。例 M1：`curl -X POST /register ... && curl -X POST /login ...` 拿到 token。

### 问题 D2：上线成功标准缺失
- **判断**：过度（漏关键）
- **守则**：第 4 条
- **理由**：PRD 没定义「上线成功」=什么。
- **建议**：写一条 happy path：「拿一个新 CDK，在新浏览器走完 注册→看到今日新闻→关掉浏览器→重开能继续看」=上线成功。

### 问题 D3：防爬效果 KPI 缺失
- **判断**：合理但不必要量化
- **守则**：第 4 条
- **理由**：PRD §5.4 已经反思过「防爬投入 = 月卡价 × 盗用率」，这个判断比硬指标更适合个人项目。
- **建议**：不强求 KPI。
