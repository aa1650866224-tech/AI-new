# 数据源重构 设计文档

**日期**：2026-05-08
**项目**：AI 新闻日报助手
**性质**：Breaking Change（数据结构、前端、采集层全部改动）

---

## 1. 背景与动机

当前项目从 6 个源（X / HackerNews / Reddit / GitHub / ProductHunt / 量子位 RSS）采集 AI 资讯，存在三类问题：

1. **覆盖维度结构性缺失**：没有官方厂商博客（OpenAI/Anthropic 等）作为一手发布源、没有政策监管动态、缺少一线开发者深度内容、中文媒体仅量子位一家。
2. **现有源信噪比偏低**：HackerNews 首页热门含大量非 AI 内容、Reddit 的 r/artificial 是新手社区、r/MachineLearning 近年水化、X 大 V 推文未被白名单过滤。
3. **设计哲学错位**：当前用"综合精选 Top 15 + 6 个源 tab"的模式，本质是 RSS 聚合器，不是日报。读者的精选权被 LLM 的 importance 分层算法替代。

经资深编辑视角审视后，方向调整为：**减少源数量、提高每条密度、按读者动作切板块**。

## 2. 设计决策（已确认）

### 2.1 数据源最终清单（22 个源）

#### 保留并改造（5 个）

| 源 | 改造点 |
|---|---|
| **X** | 维护 ≤30 人白名单（Karpathy / Jim Fan / Sam Altman / Aran Komatsuzaki / Simon Willison / Yann LeCun / Greg Brockman 等），只抓含外链或 quote tweet 的推文。**通过 RSSHub 抓取，不再走官方 API**。 |
| **HackerNews** | 加严过滤：`comments >= 150 AND title 含 [ai, llm, gpt, claude, anthropic, openai, gemini, meta, model, agent]` |
| **Reddit** | subreddit 列表砍到只剩 `r/LocalLLaMA`，删除 `r/MachineLearning` 和 `r/artificial` |
| **GitHub Trending** | 完全保留现有 7 天滚动雷达 + verdict 4 档体系（true_use/hype_only/marketing/abandoned）。这是项目最成熟资产，不动。 |
| **量子位 RSS** | 不动 |

#### 新增（17 个源）

| 类别 | 源清单 | 抓取方式 |
|---|---|---|
| **海外厂商博客**（4） | OpenAI / Anthropic / Google DeepMind / Meta AI | RSS 直订 |
| **国内厂商博客**（4） | DeepSeek / 智谱 / Kimi / 通义 | **Firecrawl**（官网为 SPA，无 RSS） |
| **政策监管**（3） | 网信办 cac.gov.cn / EU AI Act 跟踪站 / 美国 White House AI EO 公告 | **Firecrawl**（部分 RSS 可补） |
| **一线开发者 blog**（4） | Simon Willison / Lilian Weng / Eugene Yan / Chip Huyen | RSS 直订 |
| **中文媒体补强**（1） | 机器之心（jiqizhixin.com） | RSS 直订 |
| **HuggingFace**（1） | HF Trending Models | HF API |

#### 明确不加 / 砍除

- **X 旧 collector** 直接删除（不留死代码）
- **ProductHunt** 砍除（信号已被厂商博客覆盖、偏 SaaS 营销）
- **HuggingFace Daily Papers** 不加（伪精选，arxiv 当天就在 X 上被转过）
- **TLDR AI** 不加（选题偏硅谷 B 端，对中国开发者不对口）
- **新智元** 不加（标题党比量子位重）
- **r/MachineLearning** / **r/artificial** 删除

#### 工具栈分工

| 工具 | 用途 | 成本 |
|---|---|---|
| RSS 直订 | 海外厂商 + 个人 blog + 机器之心（共 9 个 RSS 源） | $0 |
| RSSHub | X 白名单（替代官方 API） | $0（自部署 / 公开实例） |
| Firecrawl | 国内厂商 4 + 政策站 3 = 7 个 SPA 站点 | **Free tier（500 credits/月）** ，~200 次/月，零成本 |
| HuggingFace API | HF Trending Models | $0 |

### 2.2 板块结构（4 个 Tab）

| Tab | 内容 | 显示策略 | 每日条数硬上限 |
|---|---|---|---|
| **今早必读** | 海外厂商博客 + 国内厂商博客 + 政策动态 + 机器之心/量子位头条 | 每日 | 8 条 |
| **圈子在吵** | X 白名单 + HN（严过滤）+ r/LocalLLaMA + 4 个个人 blog | 每日 | 10 条 |
| **GitHub 雷达** | 7 天滚动 + verdict 4 档体系 | 每日 | 不变（保持现有逻辑） |
| **周末再看** | HF Trending Models + arxiv 精选（X 兜底）+ 长文 | **全天可访问**，周日有更新提示 | 不设硬上限 |

**关键约束**：

- **综合精选 Top 15 完全砍除**——不再让 LLM 替读者排序
- **GitHub 雷达独立成 tab**——不挪到"周末再看"，保留它作为日常基础设施的地位
- **"周末再看"全天可访问**——内容偏沉淀型，但不强制周日才显示

---

### 2.3 输出形态调整

#### 编辑按语（新增字段 `editor_note`）

- LLM 在 summarize 阶段为每条生成 80-120 字的"为什么这条值得读"
- 不复述 chinese_summary，必须提供独立判断（背景、影响、对比）
- **不做人工审核**——LLM 出什么是什么，相信模型 + 简化流程

#### `importance` 字段彻底砍除

- 当前非 GitHub 源都有 `importance ∈ {重磅, 值得关注, 了解即可}`
- 该字段唯一作用是给综合精选分层；既然综合精选砍了，字段一并删
- LLM prompt 移除该字段、JSON 输出移除该字段、前端不再读

#### 翻译策略分级

| 源类别 | 翻译策略 |
|---|---|
| 厂商博客 / 个人 blog / 政策 / 中文媒体 | 全文翻译（深度内容） |
| HN / Reddit / X | 仅翻译标题 + AI 摘要 + 前 200 字（讨论类，原文链接为主） |
| 政策动态 | 全文翻译 + 关键术语保留英文（如 "AI Act"、"Section 230"） |
| GitHub | 维持现有 README 抓取 + verdict 摘要逻辑 |

## 3. 数据契约变化（Breaking Change）

### 3.1 顶层 JSON 结构

```json
// 旧
{
  "date": "...",
  "generated_at": "...",
  "overview": "...",
  "count": 15,
  "items": [ /* 综合精选 Top 15 */ ],
  "by_source": {
    "X": [...], "HackerNews": [...], "Reddit": [...],
    "GitHub": [...], "ProductHunt": [...], "量子位": [...]
  }
}

// 新
{
  "date": "...",
  "generated_at": "...",
  "overview": "...",
  "by_section": {
    "morning":    [/* 今早必读，≤8 */],
    "discussion": [/* 圈子在吵，≤10 */],
    "github":     [/* GitHub 雷达，保持现有 verdict 分组结构 */],
    "weekend":    [/* 周末再看，无上限 */]
  }
}
```

### 3.2 单条目字段变化

**所有源**：
- ✅ 新增 `editor_note: string`（80-120 字，LLM 生成）
- ❌ 删除 `importance`
- ✅ 新增 `section: "morning" | "discussion" | "github" | "weekend"`（板块映射，便于检索）

**GitHub 源**：现有字段（`verdict_tag` / `verdict.{category_tag, who_should_care, prerequisites, similar_projects}` / `radar_date`）全部保留。

### 3.3 历史数据迁移策略

**选定方案 c：归档旧 JSON，新结构干净起步。**

- 现有 `data/daily/*.json` 移至 `data/daily/_archive/`
- 现有 `web/data/*.json` 同步移至 `web/data/_archive/`
- `web/data/index.json` 重建，只列新结构日期
- 前端不做新旧兼容，专注新结构

理由：个人项目，工程量优先于历史完整性；旧数据归档保留可读，但不再展示。

## 5. 不做什么（YAGNI）

明确**不在本次重构范围**的事项，避免范围蔓延：

- ❌ 不实现 editor_note 的人工审核 dashboard
- ❌ 不做新旧 JSON 结构的前端兼容（旧数据归档不再展示）
- ❌ 不做"周日才显示周末再看"的时间逻辑（编辑原方案，被用户驳回）
- ❌ 不调整 GitHub 雷达内部逻辑（verdict / 7 天滚动 / 4 档分组全部保留）
- ❌ 不实现 RSSHub 自部署的 Docker 编排（用户自行部署或公开实例，不在脚本范围）
- ❌ 不做 X 推文的 quote tweet / 含外链过滤（先靠 RSSHub 抓全量 + 后续手工调白名单）—— 后续可加
- ❌ 不实现政策动态的"重要级别打标"（先全量收，看效果再加分级）
- ❌ 不做 ProductHunt 的迁移（直接砍，不归档）
- ❌ 不做 keywords.yaml 的调整（继续做现有过滤）

## 6. 风险与缓解

| 风险 | 缓解 |
|---|---|
| Firecrawl free tier 超额 | 7 站 × 30 天 = 210 次/月，远低于 500 上限；超了再升级 Pro |
| RSSHub 公开实例不稳定 | 优先自部署（Docker 一行命令）；公开实例作为 fallback |
| 国内厂商博客 SPA 结构变化导致 Firecrawl 抓不到 | 每个站点单独写一份 prompt 引导 Firecrawl 提取（标题/时间/正文）；监控空结果率 |
| HuggingFace API 限流 | Trending Models 量小（每天 ~50 个），单次调用即可，无限流压力 |
| editor_note 质量参差 | 用户已确认不做审核；如低质率高，后续再加质量阈值 |

## 7. 验收标准

重构完成后，应满足：

1. ✅ `python src/main.py` 跑完一次，不报错
2. ✅ 生成的 `data/daily/YYYY-MM-DD.json` 包含 `by_section` 而非 `items`/`by_source`
3. ✅ 4 个板块各自有内容（除非该日某板块确实空）
4. ✅ `morning` 不超过 8 条，`discussion` 不超过 10 条
5. ✅ 每条都有 `editor_note` 字段，长度 60-150 字
6. ✅ 没有任何条目带 `importance` 字段
7. ✅ GitHub 雷达 tab 在前端展示与重构前**视觉一致**（verdict 4 档分组样式不变）
8. ✅ 旧 JSON 已归档至 `data/daily/_archive/` 与 `web/data/_archive/`
9. ✅ `web/data/index.json` 只列新结构日期
10. ✅ 前端 4 个 tab 均可正常切换、内容渲染正常、editor_note 可见
